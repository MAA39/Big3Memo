# Phase 4: 検証フェーズ (Verification)

## ❌ やったこと (What We Did)
**「シミュレータ間通信のデバッグ」**
*   iPhoneシミュレータでSaveを押し、Watchシミュレータで待つ。
*   反応がないため、コードを見直し、ログを埋め込み、再ビルドする……という長いループ（wait time）に入った。
*   結果として「動いている画面」を見る機会が極端に減り、モチベーションが低下した。

## 💥 失敗の要因 (Failure Analysis)
**「シミュレータを信用しすぎた (Trusting the Simulator)」**
*   WatchConnectivityは、シミュレータ環境では頻繁に接続が切れたり、そもそもペアリングが認識されなかったりする（Xcodeの仕様）。
*   コードが正しくても動かない状況に対し、コードを疑って修正を加えてしまい、逆にバグを生む可能性があった（迷走）。

## ✨ 理想の進め方 (Success Path)
**「デバッグ機能（Backdoor）の早期実装」**
1.  **Debug Button**: まだ通信機能ができていない段階で、Watchアプリに「デバッグ: サンプルデータをロード」というボタンを配置する（今回最後にやったこと）。
2.  **Bypass Connectivity**: 通信が死んでいても、このボタンさえ押せばトレーニング機能の開発・確認が進められるようにする。
3.  **Physical Device**: 通信のテストは、シミュレータで粘らず、最初から実機（自分のiPhoneとApple Watch）で行う。実機の方が圧倒的に安定している。

**教訓:**
> 「もし通信が完全に壊れても、開発を続けられるか？」と常に自問し、抜け道を用意しておく。
